SAA + OpenAI Realtime

This demo wires @attenlabs/saa-js to an OpenAI Realtime session. SAA gates which speech reaches the model — background media audio, side conversations, and your own voice during playback are filtered out before tokens are spent. Tool calling, transcripts, and barge-in are wired through.

    How it works

    Three connections run concurrently:

    1. SAA WebSocket — receives mic + webcam, returns speechReady events with PCM16 @ 16 kHz audio for utterances classified as device-directed (cls 2).
    2. OpenAI Realtime WebSocket — receives audio appended via input_audio_buffer.append (PCM16 @ 24 kHz) and emits text + audio responses.
    3. Local audio playback — plays the assistant’s response. While audible output is active, wrap playout with client.beginAgentOutput(); hard-suppressed speechReady is not the barge-in signal. Use the provider interruption/cancel path or SAA x-saa.interruptCandidate when enabled.

    Per-utterance dataflow:

    mic → SAA → speechReady (PCM16 16 kHz)
            → resample 16→24 kHz
            → OpenAI input_audio_buffer.append
            → OpenAI response.create
            → response.audio.delta (PCM16 24 kHz)
            → AudioContext playback
  barge-in: SAA emits speechReady mid-playback → response.cancel
    Why we resample 16→24 kHz

    SAA’s speechReady audio is 16 kHz mono PCM16 (the SAA server’s native rate). OpenAI Realtime’s pcm16 format is 24 kHz mono. Skipping the resample makes the model hear pitch-shifted, 1.5x-fast audio — transcripts degrade noticeably and tool-call accuracy drops. The conversion is one linear-interpolation pass per utterance.
